# Roles: Onboarding & Handoff Checklist

Purpose
- Provide concrete onboarding, handoff, and responsibility-checklist templates for common personas to reduce ambiguity and accelerate onboarding.

How to use
- Copy the relevant checklist into a project README or the project one-pager.
- Use as acceptance criteria for role handoffs and readiness for release.

## Generic Role Onboarding Checklist
- [ ] Role owner named in project one-pager
- [ ] Role introduction meeting scheduled with team
- [ ] Access & permissions granted (repos, project boards, dashboards)
- [ ] Role-specific systems and runbooks shared
- [ ] Initial responsibilities and expectations documented
- [ ] Communication cadence and escalation path confirmed

## Developer Onboarding (example)
- [ ] Assigned to first 1–2 tickets with clear acceptance criteria
- [ ] Local dev environment and CI access confirmed
- [ ] Code style, PR and branching conventions reviewed
- [ ] Paired review with an experienced team member scheduled

## QA Onboarding (example)
- [ ] Test environments and test data access confirmed
- [ ] Test reporting and defect triage flow documented
- [ ] Automation ownership and CI integration explained
- [ ] Release validation checklist reviewed

## UX Onboarding (example)
- [ ] Design system and component library access confirmed
- [ ] Handoff format (Figma/Sketch/Zeplin) and annotations standard explained
- [ ] Usability test plan templates shared

## Technical Writer Onboarding (example)
- [ ] Docs repo structure reviewed (docs/, .copilot/, README)
- [ ] Release notes template shared
- [ ] Process for publishing/updating docs explained

## Support / Customer Success Onboarding (example)
- [ ] Incident triage flow and ticketing access
- [ ] Knowledge base edit rights and templates
- [ ] Runbook for high-impact features shared

## Handoff / Release Readiness Checklist (cross-role)
- [ ] Acceptance criteria met and tested
- [ ] QA sign-off on release candidate
- [ ] Documentation (end-user + runbooks) updated
- [ ] Support notified and provided with triage steps
- [ ] Rollback and mitigation plan documented
- [ ] Post-release monitoring and owners assigned

Notes
- Include these checklists in project one-pagers or release notes as needed.
- Keep checklists lightweight; update them as team practices evolve.
